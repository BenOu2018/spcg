# AGENTS.md

本文件是 agent/AI 在本项目中开发新功能、修复问题、重构代码前的第一入口。

如果本文件与临时对话指令冲突，优先遵守用户在当前任务中的明确要求；如果本文件与代码事实冲突，先读取现有实现并说明差异，不要凭空改写系统。

## 第一入口

- 开发新功能前，先读本文件。
- 功能范围不明确时，先读 `DOCS_INDEX.md` 找到对应业务、架构、内容或 UI 文档。
- 涉及世界观入口、算法遗迹、`/world` 页面时，必须读 `world.md`。
- 涉及工程分层、API、Server Actions、service、repository、数据库访问时，必须读 `docs/architecture-boundaries.md`。
- 涉及题库、关卡、内容生产时，按 `DOCS_INDEX.md` 指向读取 `problem-bank/README.md`、`problem-bank/STRUCTURE.md`、`problem-bank/AGENT_BRIEF.md` 等相关文档。

## 开发前必读

开始修改前必须先理解：

- 当前功能入口在哪里。
- 数据从哪里来，写到哪里去。
- 是否已有 service、repository、组件、类型、脚本或资产流程可复用。
- 该功能面向普通用户、老师、管理员还是运营后台。
- 本次修改的最小安全范围是什么。

不要先写代码再回头找约束。不要因为实现快，就绕过已有架构。

## 实现原则

- 如果你准备使用低效实现，请先停止，并说明为什么不用更优方案。
- 请按“生产环境可承载大量数据的后台管理页面”标准实现，而不是 demo 页面。
- 默认选择可维护、可扩展、可验证的实现，而不是一次性最快实现。
- 优先复用现有类型、服务、组件、样式模式和资产管线。
- 不复制平行系统；除非用户明确要求，否则不要为同一业务再造一套页面、状态或数据流。
- 不为短期方便引入长期技术债。确实需要折中时，必须说明原因、影响范围和后续替换路径。

## 性能与大量数据标准

凡是列表、后台、数据管理、老师端、运营端、审核端、排行榜、提交记录、学生数据、题目数据，都默认按大量数据处理：

- 服务端分页，不要一次性全量拉取后在前端分页。
- 查询应支持必要的筛选、搜索、排序和稳定分页。
- 大表查询要考虑索引、where 条件、limit、offset 或 cursor 策略。
- UI 必须有加载态、空状态、错误态和权限不足状态。
- 避免在 React render 中做重计算；必要时做预处理、缓存、分页或虚拟化。
- 避免无边界轮询、无节制并发请求和重复请求。
- 不在前端硬编码真实业务数据作为长期实现。
- 图片、视频和大资产必须走现有 public asset/压缩/引用流程，不把大文件随意塞进页面代码。

如果需求看起来可以用一个简单数组、全量查询、临时 mock、前端过滤快速完成，也要先判断它是否会在生产数据量下失效。

## 后台管理页面标准

后台管理页面不是 demo。实现时默认包含：

- 清晰的信息架构和可扫描布局。
- 分页、筛选、搜索、排序的可扩展设计。
- 批量数据场景下稳定的响应和明确的状态反馈。
- 权限边界：管理员、老师、学生的数据访问必须隔离。
- 操作确认、失败提示、成功反馈和可恢复路径。
- 表格或列表字段要服务真实运营判断，不堆无意义信息。
- 移动端可用，但后台优先保证桌面密集信息处理效率。

禁止只为了看起来完成而写静态卡片、假数据列表、一次性全量加载或无法承载真实数据的交互。

## 架构边界

遵守 `docs/architecture-boundaries.md`：

- UI 组件只负责渲染、收集交互和调用批准的 action/API。
- UI 不直接 import `@/lib/db`、`@/lib/repositories/*` 或 `@/lib/services/*`。
- Server Actions 和 API Routes 负责 auth、validation、response shape、revalidation，并调用 service。
- Service 负责业务流程。
- Repository 负责 SQL 和事务相关的数据访问。
- `db` 只负责 PostgreSQL pool 和底层 query helper。
- 新代码不要扩大历史 allowlist；碰到旧代码时，优先向 service/repository 分层迁移。

涉及权限时必须使用已有的 `requireAdmin()`、`requireTeacher()`、`requireTeacherOwnsStudent()` 等机制，不要只在 UI 上隐藏按钮。

## UI 与资产规则

- 优先沿用当前页面的设计系统、CSS module、组件结构和视觉语言。
- 不做营销页式空壳；应用页面要直接服务真实任务。
- 后台和运营工具要安静、密集、可扫描，不要过度装饰。
- 游戏化页面可以有氛围，但关键文本、按钮和状态必须清晰可读。
- 关键中文文案尽量用 HTML/CSS 渲染，不烘进生成图片，避免文字错误和后续不可维护。
- 新增图片资产放入合适的 `assets/` 目录，并通过现有脚本同步到运行时 public 目录。
- 不随意修改全局样式；能局部隔离就使用局部样式。

## world 线特殊约束

`world.md` 是算法遗迹世界观入口的事实来源。

第一轮 world 线默认边界：

- `/world` 只负责品牌入口和情绪建立。
- `/world -> /map -> /level/[id]` 是当前进入路径。
- 不额外维护 `/world/map` 或 `/world/level/[id]` 平行路线。
- 原 `/map`、`/level/[id]`、题库、判题、进度系统保持复用。
- 未明确要求前，不改原题目、题单、章节和 OJ 主流程。

任何会把 world 线扩展成第二套系统的实现，都必须先停下来说明原因和风险。

## 验证要求

修改后按影响范围运行检查：

- 仅改文档：检查文档内容和 `git status --short --untracked-files=all`。
- 改前端页面或组件：至少运行 `npm run web:typecheck`。
- 改架构分层、actions、services、repositories：运行 `npm run check:architecture`，必要时运行 `npm run check`。
- 改资产引用：运行 `npm run prepare:public-assets`，并确认运行时路径可访问。
- 改题库或内容导入：运行对应的 dry-run 校验脚本。

如果检查无法运行，必须说明原因、已做的替代验证和剩余风险。

## 禁止事项

- 不要为了速度写会在真实数据量下崩掉的实现。
- 不要全量读取大数据后在前端过滤、排序、分页。
- 不要绕过 service/repository 分层直接访问数据库。
- 不要只做 demo 态、无加载态、无错误态、无权限边界的后台页面。
- 不要重写用户已有改动，不要用 destructive git 命令回滚工作区。
- 不要把无关重构混入当前任务。
- 不要在没有明确需求时引入新框架、新状态库或新数据模型。
- 不要把关键业务逻辑藏在 CSS、静态文案或临时 mock 中。
